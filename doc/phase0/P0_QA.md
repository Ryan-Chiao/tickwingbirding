## T0-8 QA 测试策略文档（03-20 ~ 03-21）⭐ 主要任务

### 工作内容

编写 `doc/testing/TestStrategy.md`，包含以下章节：

#### 1. 测试范围

按 Phase 定义测试范围：

| Phase | 测试重点 |
|-------|---------|
| Phase 1 | 注册、登录、Token 刷新、密码找回、资料编辑、安全基线 |
| Phase 2 | 视频上传（含规格校验）、转码、播放、点赞、评论、可见性、个人主页 |
| Phase 3 | 关注、Feed、搜索、通知、举报 |
| Phase 4 | 管理后台 CRUD、审核流程、统计准确性 |
| Phase 5 | 设备绑定/解绑、API Key 鉴权、心跳、设备上传、归属逻辑 |
| Phase 6 | 移动端核心功能、推送通知、崩溃率 |

#### 2. 测试分层与工具

| 测试类型 | 职责 | 工具 | 占比建议 |
|---------|------|------|---------|
| 单元测试 | 业务核心逻辑（Service 层） | Jest | 60% |
| 集成测试 | DB / 缓存 / 队列 / 对象存储联动 | Jest + Testcontainers | 20% |
| E2E 测试 | 主链路端到端验证 | Playwright (Web) / Detox (Mobile) | 10% |
| 性能测试 | 关键 API 压测 | k6 / Artillery | 5% |
| 安全测试 | 鉴权、注入、越权、上传漏洞 | OWASP ZAP / 手动渗透 | 5% |

#### 3. 覆盖率目标

| 模块 | 最低覆盖率 |
|------|---------|
| auth / user / video（核心） | ≥ 80% |
| feeder / community / notification | ≥ 70% |
| admin | ≥ 60% |
| 工具函数 / 配置 | 不做硬性要求 |

#### 4. E2E 关键用例清单（Phase 1-2）

至少列出以下用例（不需要写脚本，只需列出用例名和步骤概要）：

**Phase 1：**
1. 用户注册 → 收到验证邮件 → 验证成功 → 登录
2. 用户登录 → 获取 Access Token → Token 过期 → Refresh → 获取新 Token
3. 用户登录 → 修改资料（昵称/头像/简介）→ 查看主页验证更新
4. 忘记密码 → 发送重置邮件 → 重置成功 → 用新密码登录
5. 未认证请求 → 返回 401

**Phase 2：**
6. 用户上传视频 → 转码完成 → 在视频列表中可见 → 可播放
7. 上传超过 300MB 的视频 → 被拒绝
8. 上传超过 2 分钟的视频 → 被拒绝
9. 设置视频为 private → 其他用户搜索不到该视频
10. 用户 A 点赞视频 → 点赞数 +1 → 再次点赞（幂等）→ 点赞数不变
11. 用户评论视频 → 评论可见 → 回复评论 → 二级回复可见
12. 视频转码失败 → 重试（最多 3 次）→ 仍失败则标记 failed

#### 5. 缺陷分级标准

| 等级 | 定义             | 示例                 | 修复时限        |
| -- | -------------- | ------------------ | ----------- |
| S1 | 系统不可用 / 数据丢失   | 注册崩溃、视频上传后文件丢失     | 4h 内修复      |
| S2 | 核心功能不可用        | 播放失败、无法评论、登录异常     | 24h 内修复     |
| S3 | 功能受限但有 workaround | 排序不生效、缩略图显示错误、分页异常 | 当前 Sprint 内 |
| S4 | 体验问题           | 文案错误、样式偏移、动画卡顿     | 排入 backlog  |

#### 6. 缺陷流转流程

```
新建 (New) → 确认 (Confirmed) → 修复中 (In Progress) → 待验证 (Fixed)
                                                         ↓
                                                   验证通过 → 关闭 (Closed)
                                                   验证不通过 → 重新打开 (Reopened)
```

#### 7. 发布门禁标准

| 环境 | 门禁要求 |
|------|---------|
| dev → stage | lint + typecheck + 单元测试全部通过 |
| stage → prod | 上述 + E2E 主链路通过 + S1/S2 缺陷清零 + smoke test 通过 |

#### 8. 自动化策略

| 优先级 | 自动化内容 | 阶段 |
|-------|---------|------|
| P0 | 单元测试（随开发同步编写） | Phase 1 起 |
| P0 | CI 流水线中集成测试 | Phase 1 起 |
| P1 | E2E 主链路测试 | Phase 2 起 |
| P1 | 性能基线测试（Feed、播放） | Phase 2 起 |
| P2 | 安全扫描 | Phase 3 起 |

### 条件约束
- Phase 0 只需产出策略文档，不需要写测试代码
- 实际测试脚本从 Phase 1 开始随开发同步编写
- 测试环境应与开发环境隔离（至少数据库实例不同）

### 注意事项
- 考虑 Testcontainers 做集成测试的数据库隔离，每个测试用例独立 schema
- 安全测试不需要 Phase 0 就规划细节，但需要在策略文档中占位
- 性能测试的基线数据需要在 Phase 2 首次建立，后续每个 Phase 对比
- 视频相关测试需要准备测试视频素材（不同分辨率、时长、编码），建议放在测试专用的对象存储 bucket
